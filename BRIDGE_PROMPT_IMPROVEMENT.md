# Agent Bridge Prompt 改进建议与执行计划

## 现状分析

当前 `orchestrateLongTask` 函数中使用的 prompt 存在以下问题：

1. **边界定义不够清晰**：虽然有基本规则，但缺乏明确的执行边界和失败行为模式
2. **工程约束不足**：缺少关于代码质量、安全性和性能的具体约束
3. **优先级体系不明确**：没有明确的基线约束与补充说明的关系
4. **记忆治理协议缺失**：没有指导模型如何处理记忆信息
5. **缓存效率优化不足**：没有明确的静态指令与动态上下文的分离
6. **验证闭环不完整**：没有明确的"完成"定义和验证要求

## 改进建议

### 1. 确立"控制平面"思维

- **定义明确的执行边界**：
  - 明确禁止越权操作（如修改 `.git` 目录或工作区外文件）
  - 定义失败行为模式（如解析失败、应用失败时的重试策略）
  - 明确报告责任（如详细记录错误信息和执行状态）

- **写入工程约束**：
  - 禁止在无必要时制造抽象
  - 要求保持代码风格一致性
  - 强制进行错误处理和边界情况检查

### 2. 采用分层与模块化架构

- **职责分离**：将 prompt 拆分为以下模块：
  1. **身份定义**：明确模型的角色和核心职责
  2. **系统级规则**：执行边界、安全约束和失败行为
  3. **任务约束**：基于操作类型的具体要求
  4. **输出纪律**：格式要求和验证标准

- **动态注入**：支持将以下内容作为动态段落注入：
  - 反馈历史
  - 工作区上下文
  - 操作类型约束
  - 任务特定说明

### 3. 建立清晰的优先级体系

- **区分基线与补充**：
  - 基线：核心规则和约束，适用于所有任务
  - 补充：特定任务或操作类型的额外说明
  - 确保补充说明在基线之上叠加，而非替换

- **优先级排序**：
  1. 安全约束（最高）
  2. 执行边界
  3. 输出格式要求
  4. 代码质量标准
  5. 任务特定说明（最低）

### 4. 规范记忆与知识治理协议

- **指导记忆形成**：
  - 明确哪些信息值得存入长期记忆（如架构决策、关键 API 使用）
  - 建议使用结构化格式（如 Frontmatter）存储记忆
  - 指明哪些信息不应占用记忆空间（如临时计划、具体实现细节）

- **区分入口与细节**：
  - 保持记忆索引简洁，将细节推向具体的专题文件
  - 建立记忆检索机制，确保相关信息可被有效利用

### 5. 优化缓存与计算效率

- **显式划分缓存边界**：
  - 将相对稳定的静态指令（如身份定义、系统级规则）与频繁变化的上下文（如任务指令、反馈历史）分开
  - 使用明确的边界标识，提高 prompt 缓存的命中率

- **动态上下文优化**：
  - 根据任务难度和相关文件动态调整上下文内容
  - 优先包含与任务直接相关的文件和信息

### 6. 强制执行验证与闭环

- **明确"完成"的定义**：
  - 代码变更已成功应用
  - 语法和语义验证通过
  - 没有未解决的错误或警告
  - 符合所有工程约束

- **独立验证要求**：
  - 要求模型通过独立验证（如运行测试）来证明有效性
  - 禁止仅依靠模型自述"已完成"作为任务完成的依据

## 执行计划

### 第一阶段：重构 `adapter.buildPrompt` 函数

1. **模块化 prompt 构建**：
   - 将 prompt 构建逻辑拆分为多个函数，每个函数负责一个模块
   - 实现动态注入机制，支持不同类型的上下文和约束

2. **增强系统级规则**：
   - 添加明确的执行边界和安全约束
   - 定义详细的失败行为模式和报告要求

3. **优化缓存策略**：
   - 实现静态指令与动态上下文的分离
   - 添加缓存边界标识，提高缓存命中率

### 第二阶段：改进 prompt 内容

1. **更新身份定义**：
   - 明确模型的角色为"代码编辑代理"，强调专业性和可靠性
   - 突出核心职责：安全、准确、高效地完成代码变更

2. **完善系统级规则**：
   - 添加详细的工程约束和代码质量标准
   - 明确禁止的操作和行为

3. **优化任务约束**：
   - 基于操作类型提供更详细的指导
   - 添加针对不同难度任务的特定要求

### 第三阶段：集成与测试

1. **集成到 `orchestrateLongTask` 函数**：
   - 更新函数调用，使用新的 prompt 构建机制
   - 确保与现有的任务执行流程兼容

2. **测试与验证**：
   - 运行现有的测试套件，确保功能正常
   - 执行端到端测试，验证 prompt 改进的效果
   - 收集反馈并进行必要的调整

### 第四阶段：文档与维护

1. **更新文档**：
   - 记录新的 prompt 结构和设计原则
   - 提供使用指南和最佳实践

2. **建立维护机制**：
   - 定期审查和更新 prompt 内容
   - 收集用户反馈，持续改进

## 预期效果

通过以上改进，预计将：

1. **提高代码生成质量**：更准确、更符合工程标准的代码变更
2. **增强系统安全性**：减少越权操作和不安全的代码变更
3. **提高执行效率**：通过缓存优化和上下文管理，减少 token 使用和执行时间
4. **增强可靠性**：明确的验证闭环和失败处理机制，提高任务成功率
5. **便于维护和扩展**：模块化架构和清晰的优先级体系，便于后续功能扩展
